# slide-infographic Skill（Marp スライドにインフォグラフィックを差し込む）

> Codex 用の派生スキルです。正本は `docs/agent-instructions/skills/slide-infographic/SKILL.md.liquid`。`bun run sync:agent-docs` で生成するため、直接編集しないでください。

> **Core Principle**: スライド本文（見出し・箇条書き・表）は内容を変えない。ページに「読む前の地図」となる1枚絵を添えるだけ。図解は **gpt-image-2 で生成した「手書き風イラスト＋正確な日本語テキスト」画像**。Markdown 資料と違いスライドは 1 ページの領域が固定なので、**差し込み後に必ず `bun run build && bun run check` ではみ出しを機械検知し、画像を入れたページだけ目視する**。

## 姉妹スキルとの使い分け

| スキル | 媒体 | 出力 |
|---|---|---|
| gallery-image-gen | 汎用（画像単体） | gallery/ に画像を置いて公開 |
| **slide-infographic（本スキル）** | Marp スライド（slides/*.md） | **スライドへの画像記法追記 + gallery/ への画像配置** |

> **共通エンジン**: 画像生成の呼び出し作法（組み込み image_gen・API 課金回避・1枚ごとのコピー確定）は **`../gallery-image-gen/SKILL.md` に従う。本スキルを使うときは必ず先にそちらも読むこと。**「画像内に文字を入れるなら逐語の正確な文字列を渡して文字化けを防ぐ」原則も共通。**文字は多めでよい**（逐語で渡せば高密度でも崩れない。「文字は少なめに」は誤り）。

## 全体ワークフロー

1. **対象デッキの特定**: `slides/<YYYY-MM-DD>-<slug>.md` をユーザーから受け取る(IDE で開いているファイル・直前に作ったデッキが文脈で明らかならそれ)
2. **ページ構成の把握**: `---` 区切りでページを数える。**全ページに入れない**。lead の章扉・表紙・まとめは原則スキップし、「文字が多く、図があると理解が早くなる本文ページ」に絞る。**対象ページはユーザーの依頼範囲が正**（依頼に含まれないページへ勝手に広げない。範囲が曖昧なら候補を提示して合意を取る）
3. **画像プランの提示**: 「どのページに・どのレイアウトで・何を描くか」の一覧を短く出して合意を取る
4. **ページごとにプロンプトを起こす**: 下記【スタイル定型】＋【そのページの正確なテキスト】。箇条書き・表・ラベルは**本文から逐語コピーして文字列で渡す**
5. **1枚試作 → 合意 → 残りを生成**: いきなり全ページ作らない。生成は gallery-image-gen の「画像生成」に従う（1枚試作は単発実行、合意後の残りは「複数枚の並列一括生成」で並列に）
6. **レビュー（必須）**: 生成画像を1枚ずつ開いて、文字化け・誤字・内容違いを確認。崩れていたらその1枚だけ再生成
7. **gallery へ配置**: `gallery/<YYYYMM>-<デッキslug>/<内容slug>.png`（**ページ番号をファイル名に入れない** — ページの追加・削除・並び替えで番号がすぐズレる）
8. **スライドへ差し込み**: 下記「差し込みパターン」の Marp 記法で該当ページに 1 行追加。**本文の文章・表は原則触らない**（例外は「はみ出し調整」の節）
9. **ビルドしてはみ出し確認（必須）**: `bun run build && bun run check` で機械検知 → `dist/slides/<デッキ>.pdf` で画像を入れたページだけ目視。画像のトリミング・文字との被り・改行崩れがないか確認する
10. commit / push はユーザーの指示があってから（push = 公開）

## 画像の置き場所と参照

- 置き場所: `gallery/<YYYYMM>-<デッキslug>/`（例: `gallery/202607-ai-tool-strategy/`）。gallery/README.md の命名規則に従う
- ファイル名: `<内容がわかる英語slug>.png`（例 `three-companies.png`）。**ページ番号を入れない**（ページの増減で番号がズレて対応が壊れる。どのページで使っているかはスライド md 内の参照行で追える）
- スライドからの参照は**相対パス**: `![...](../gallery/<YYYYMM>-<デッキslug>/<slug>.png)`
  - ビルド時、slides/*.md → dist/slides/*.html、gallery/ → dist/gallery/ にコピーされるため、`../gallery/...` は**リポジトリ内でも公開サイト上でも同じ相対関係で解決する**（ロゴの `../assets/brand/...` と同じ仕組み）
- 副産物として、同じ画像がギャラリー一覧 https://contents.zenshin-inc.co.jp/gallery/ にも自動掲載され、直リンクも得られる

## 差し込みパターン（Marp 記法）

スライドは 1280x720（16:9）の固定領域。ページの文字量に合わせて 3 パターンを使い分ける:

1. **右カラム背景（基本形・文字が多いページ）**: 見出しの直後に
   ```markdown
   ![bg right:38%](../gallery/<フォルダ>/<slug>.png)
   ```
   本文はそのまま左に流れる。Marp の `bg` はカラム領域を cover で埋めるため、**カラム比と合わない画像は上下（または左右）が容赦なくトリミングされる**。縦長は **768x1312**（16:9でright:33%の約422:720に近い比率・16の倍数）を目安に生成し、**画像内タイトルは入れない**（スライド側に見出しがある。上端に置いたタイトルはトリミングで断片だけ残り、かえって汚くなる実績あり）。上端・下端の各 10% は「切れてもよい余白」にし、文字・キャラを端に置かない。`right:33%`〜`right:40%` で調整
2. **ページ内の図（図解がそのページの主役のとき）**: 本文を見出し＋短い箇条書きに抑え、
   ```markdown
   ![w:640 図の説明](../gallery/<フォルダ>/<slug>.png)
   ```
   横長 1536x1024 で生成し、`w:440`〜`w:640` で収まる幅に調整
3. **表紙・章扉（lead ページ）**: **原則入れない**。zenshin テーマの lead は濃紺グラデーション背景で、ベージュ紙の手書き風画像は浮く。どうしても入れる場合はユーザーと相談の上で全面背景 `![bg opacity:.25](...)` などを試し、文字の可読性を必ず確認する

- alt テキストには内容の要約を書く（`![bg right:38% 3社比較の図解](...)` のように、bg 指定と併記できる）
- 1 ページに画像は 1 枚まで。詰め込まない

## はみ出しの予防（差し込み時に必ず適用する文章量バジェット）

16:9（1280x720）で `bg right:33%` を入れると本文幅は約 730px（**全角約 30 文字/行**）になる。**後から目視で直すのではなく、差し込み時点で次のバジェットに収める**（横幅が増えても文章量は増やさず、余白と図の可読性へ充てる）:

- **導入文は書かない**（見出し＋画像が導入の役割。どうしても要るなら 1 行まで）
- 本文の上限目安: **「箇条書き 3〜4 個（各 1〜2 行）+ コールアウト 1 個（3 行以内）」または「表 4 行 + コールアウト 1 個」**
- **blockquote / callout 内で `>` の複数行やハード改行をしない**（幅が変わると改行位置も変わる。1 つの `>` に文章を流し込む）
- **表のセルは折り返さない長さにする**（1 セル 1 行。折り返すと行数が倍増する）
- 同じ主旨を導入文とコールアウトで**二重に言わない**（どちらか一方に寄せる）
- 見出しが 2 行に折り返すページは、本文をさらに 1〜2 行減らす

## はみ出しの検証（自動チェック → 絞った目視）

1. **自動判定**: `bun run build && bun run check`（`bun run ci` にも組み込み済み）。`scripts/check-overflow.ts` が全デッキ・全ページの「領域外あふれ」を headless Chrome で機械検知し、NG ページ番号と超過 px を出す。**全ページの PDF 目視は不要になった**
2. **絞った目視**: 自動チェックが検知できるのは領域外へのあふれのみ。**画像を入れたページに限り** `dist/slides/<デッキ>.pdf` の該当ページを開き、画像のトリミング・文字との重なり・フッターとの近接を確認する
3. **単体デッキの高速ループ**: 全デッキビルドを待たずに確認したいときは
   `bunx marp slides/<デッキ>.md --images png --image-scale 1 -o /tmp/check/deck.png --theme-set themes --allow-local-files --html`
   でページ単位の PNG（deck.001.png…）を出し、該当ページだけ確認する
4. あふれたら上記バジェットに沿って圧縮する（**意味・情報は削らない**。改行位置・冗長な語尾・二重表現の整理から）

## スタイル定型（gpt-image-2 プロンプトに必ず入れる）

「親しみやすいが可愛すぎない・情報多め・フロー明快」トーン。プロンプト冒頭に以下を貼る:

```
【スタイル】手書き風のやさしいフラットイラスト。少しデフォルメした親しみやすいキャラ（にっこり笑顔）。可愛くしすぎず、キャラ・小物は小さめに添えて文字情報を主役にする。落ち着いた配色で「情報が整理されて見やすい」ことを最優先。システム・Botは丸くフレンドリーに。背景はベージュの紙(#f7f2e7)。書類・スマホ・PCなどの小物もシンプルにデフォルメ。
【タイトル見出し】（ページ内の図のみ）上部にしっかりした「見出し」を置く：左に小さな丸い色アイコン＋セクション番号バッジ（例：01）、その右に大きな太字タイトル、タイトル文字の下にマーカーで引いたような太い色帯（ハイライト）を敷く。本文よりはっきり大きく、ひと目でタイトルと分かるデザインにする。※ bg right 用の縦長画像ではこの行を使わず、代わりに「タイトルは入れない。角丸の情報カード・帯を縦に2〜3段重ね、点線や矢印でつないで上から下へ視線が流れる構成にする。画像の上端と下端は各10%をベージュの余白にして、文字やキャラを端に置かない」と指示する
【フロー】手順・流れは大きな番号＋矢印で順番が一目で追えるレイアウトにする。役割・系統が分かれる場合は色で区別し、ラベルを添える。
【情報量】ギチギチに多めにする。カードの中に「太字の見出し＋箇条書き 2〜3 行」を文字で入れ、本文の見出し・補足・表・ラベルを逐語でしっかり描く。文字だけでも内容が伝わる密度にし、余白を広く取りすぎない。文字は太めで読みやすく、誤字なく正確に整理して配置する。
【禁止】星・鉛筆・絵筆・水彩の飛び散り、本文に関係ない動物などのにぎやかし装飾は描かない。リアルすぎる人物や固い事務的な雰囲気は避ける。
【サイズ】<レイアウトに応じて: 横1536 x 縦1024（ページ内の図）/ 横640 x 縦1456（bg right 用）>。
日本語は誤字なく正確に描いてください。
```

そのうえで `【描く内容：<ページタイトル>】` として、そのページの**正確なテキスト**（タイトル・箇条書きの各項目・表の各セル・ラベル）を箇条書きで渡す。表は「A｜B｜C」の区切りで列を明示すると崩れにくい。

### スタイルのキモ（ハマりどころ）

- **文字は必ず正確な文字列で渡す**。「○○について」のような曖昧指示だと gpt-image が創作・誤字する
- **ページ内の図ではタイトルの「見出し化」指定を入れる**（アイコン＋番号バッジ＋マーカー下線）。**`bg right` 用の縦長画像ではタイトルを入れない**（スライド側に見出しがあり、上端はトリミングで切れる。画像側は絵とラベル中心に）
- **【フロー】【情報量】は `bg right` 用の縦長でも省略しない**。タイトルと一緒にこれらまで削ると「マスコット1体＋ラベル数個」のスカスカな“いかにも AI が作った風”のポスターになる（2026-07 に全枚再生成になった失敗実績）。縦長では情報カードの縦積み＋点線・矢印コネクタで密度を出す
- **人物キャラ（担当者・利用者などデフォルメしたにっこり人物）も入れる**。ロボットだけだと AI 感が強まる。トーンの基準は「人物＋カード＋点線＋手描きの線」が同居する高密度な紙面——人物キャラは大きく据えず、カード・ラベルの意味を補助する脇役として複数入れる
- **可愛さは中庸**。「絵本風・ちびキャラ全開」にすると情報が減って軽くなる
- **装飾の禁止を明示**。指定しないと星・鉛筆・水彩・無関係な動物が散る
- **カタカナ長音の直後に漢数字「一」が来る語は崩れる**（「マスタ一元化」→「マスター元化」）。表記を変えて回避するか、【文字の厳守】行で正誤を明示する
- **文字は間引かない**。「スライドは縮小表示されるから」を理由に語数を絞らない。本文の箇条書き・表・補足を**逐語で**カード内に詰め込む（逐語で渡せば高密度でも崩れない）。キャラが大きく余白が広い「かわいいポスター」はトーン違い（2026-07 のユーザーフィードバック: 文字ギチギチ・文字だけでも内容が伝わる密度がこの会社の好み）

### イラスト品質の合格基準（必須）

成果物は「読みやすい図表」ではなく**手書き風イラスト入りインフォグラフィック**。文字精度を理由に HTML/CSS/Playwright/SVG で図表画像を組むのは禁止。各画像には、その資料のドメインに合った小物・キャラ（人物 / Bot・システム / PC・スマホ / 書類など）を**3 種類以上**、目に見えるイラストとして入れる。角丸カードと矢印だけの業務フロー図・スライド風に整えただけの画像も、**マスコット1体＋ラベル数個だけのスカスカな縦長ポスター**も、文字が正確でも不合格。1枚試作の確認では文字の正確さに加えて**小物・キャラが十分か・図表だけになっていないか**を見る。最終レビューでは全画像を順に開き、「同一トーンの手書き風イラスト群」に見えるか確認する。

## やらないこと / 注意

- スライド本文の文章・表・構成を書き換えない（例外は「はみ出し調整」で合意を得た最小限の調整のみ）
- 全ページへの機械的な差し込みをしない（1 ページの情報量が減り、かえって冗長になる。効くページに絞る）
- 大量のページを**確認なしに一括生成しない**。1枚合意してからバッチ
- ビルド確認（PDF 目視）を省略しない。Markdown 上で完成に見えてもスライドでは崩れていることがある
- commit / push はユーザーの明示指示があるまでしない（push = 社外公開）
