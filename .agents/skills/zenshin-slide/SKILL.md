# zenshin-slide Skill（ZENSHIN の公開スライドを作る）

> Codex 用の派生スキルです。正本は `docs/agent-instructions/skills/zenshin-slide/SKILL.md.liquid`。`bun run sync:agent-docs` で生成するため、直接編集しないでください。

> **Core Principle**: 公開デッキは **zenshin ブランドテーマ一択・16:9（1280x720）固定**。
> テーマ側の部品（callout・center・columns など）だけで組み、デッキ内に自前 CSS を書かない。
> **挿絵はデフォルトで入れない**（ユーザーの明示指示があるときのみ）。
> push = 即公開なので、commit / push はユーザーの明示指示があるまでしない。

## デッキの作り方

- ファイル: `slides/<YYYY-MM-DD>-<slug>.md`（slug は内容がわかる英語ケバブケース）
- frontmatter テンプレ:

  ```markdown
  ---
  marp: true
  theme: zenshin
  size: 16:9
  lang: ja
  paginate: true
  title: <タイトル — サブタイトル>
  description: <一覧ページに出る 1〜2 文の説明>
  footer: © ZENSHIN Inc.
  ---
  ```

  `lang: ja` は必須（テーマの `word-break: auto-phrase` による**文節単位の折り返し**が効く条件。複合語の泣き別れを自動で防ぐ）

- サイズは **16:9（1280x720）**。テーマ既定も16:9だが、デッキの意図が明示されるようfrontmatterへ`size: 16:9`を必ず書く。4:3は既存資料の互換用途に限る
- 推奨ページ構成:
  - 小型（本文 10 枚規模）: lead cover 表紙（`<!-- _paginate: false -->`）→「結論から」（center）→ 本文 → まとめ（center）→ lead contact 関連リンク
  - 大型（本文 12 枚以上・3章以上）: lead cover 表紙 →「結論から」（center）→ **［章扉（divider）→ その章の本文］を章の数だけ** → まとめ（center）→ lead contact 関連リンク
- **大型デッキは本文を書く前に3〜5章へ束ねる**: 「選び方」「比較根拠」「導入前確認」のように、読者の思考段階が変わる箇所を章の境界にする。異なる役割のページを「用途別」「その他」のような2つの大分類へ詰め込まない
- **章の現在地は必ず header ディレクティブの章ラベルで示す**: 各章の先頭ページに `<!-- header: N. 章タイトル -->` を書くと以降のページ左上に小さく表示され続ける。まとめのページで `<!-- header: "" -->` を書いてクリアする
- **章扉ページはデッキの規模で使い分ける**:
  - 本文 10 枚規模の小さいデッキでは**作らない**（テンポを分断するだけ。章ラベルで足りる）
  - 本文が **約 12 枚以上・3章以上**あるデッキでは、章が変わるたびに `<!-- _class: divider -->` の**章扉（目次兼用・濃紺背景）を必ず入れる**。最初の章にも置き、途中の章だけ省略しない。書き方: 全章の番号付きリストを置き、**現在地の章だけ太字**にする（白字＋ゴールド下線で浮き上がる）。扉ページでは `<!-- header: "" -->` でヘッダーをクリアし、直後の本文ページで `<!-- header: N. 章タイトル -->` を再設定する
  - 独立した目次ページは作らない（章扉が目次を兼ねる。lead の中扉も使わない）
- **表紙はZENSHIN HPのOGPと情報階層をそろえる**: `<!-- _class: lead cover -->`を使い、左上に`スライド | 株式会社ZENSHIN`、中央左にタイトルとサブタイトル、左下に作成日と著者、右下に白ロゴを置く。上部へ大きな空白を残して要素を下半分へ固めず、テーマの固定配置で「媒体名／主題／発行情報」の3段を明確にする。スライド一覧とOGPには1枚目のPNGが使われるため、表紙だけで内容と作成主体が判別できるようにする
- **サブタイトルへ年月・「最新版」などの日付概念を入れない**: サブタイトルは「誰が何を判断できる資料か」「どの観点で何を選ぶか」というタイトルだけでは伝わらない価値を書く。鮮度は作成日と本文の確認日で示す
- **作成日は独立したメタ情報として目立たせる**: `<div class="cover-date">作成日 YYYY.MM.DD</div>`を使い、本文より小さくてもゴールド・太字で明確に読むことができる強さにする。著者は`<div class="cover-author">氏名 <span>役職</span></div>`で続ける
- 表紙のロゴ: `<img class="cover-logo" src="../assets/brand/ZENSHIN-logo-white.webp" alt="ZENSHIN">`（右下へ固定）
- **公開デッキの最終ページは短い謝辞と連絡先にする**: `<!-- _class: lead contact -->`で「ありがとうございました」とし、その下へ質問先と関連リンクを一列で並べる。「ご清聴ありがとうございました」は登壇を前提にするため使わない。質問先は2つに限定し、[公式ホームページのお問い合わせ](https://www.zenshin-inc.co.jp/contact)と著者個人の確認済みリンクを置く。高橋俊が著者の場合は「ZENSHIN CTO 高橋 俊へのご質問」として[X `@suguru_takaha4`](https://x.com/suguru_takaha4)へ「DMまたはリプライ」と明記する。質問先2件の後に、質問経路とは分けて「ZENSHINホームページ」を関連リンクとして置く。会社と個人を別カラムへ分けず、縦方向の一つの読み筋にする。URLは画面にも表示してPDFを画像で見ても行き先が分かるようにする

## zenshin テーマの部品と書き方ルール

テーマは `themes/zenshin.css`。**部品の一覧・使い方・最新の版数はテーマ CSS 冒頭のコメントが単一情報源**（この SKILL に版数を書かない — すぐ陳腐化するため）。要点:

- **結論・メッセージ主体のページは `<!-- _class: center -->` で中央揃え**にする（「結論から」「まとめ」「推奨プラン」など）。箇条書き中心のページは左揃えのまま。center ページ内でも箇条書き・コールアウトは自動で左揃えが保たれる
- **見出しに下線・短いバーなどの装飾を付けない**（生成 AI スライドの定番装飾に見えるため、テーマからも廃止済み）
- **タイトル（h1）は全ページ中央揃え**（テーマが自動適用）で、**必ず 1 行に収める**（16:9 の目安: 全角 24〜26 文字。超えるなら言い換えで短縮する）
- **絵文字を意味のある視覚記号として使う**: 章扉は章番号・現在地・ゴールド下線だけで階層が伝わるため、タイトルへ絵文字を付けない。絵文字は3〜5項目の短い意味リストなど、項目の識別を速める箇所へ限定する。例: `- **🔑 ID・権限**`
- **リストで使う場合は同格項目の全てに付ける**: 一部の項目だけに付けると重要度の差に見えるため、兄弟項目へ同じ粒度で1つずつ割り当てる。逆に、長文リスト、比較表の各セル、全コールアウトへ反復してごちゃつかせない
- **結論文やコールアウトへ絵文字を入れない**: `GOOD / CAUTION / WARNING / INFO`は固定ラベル、中央の結論文は太字と核心語の下線が意味マーカーになるため、本文先頭の絵文字と競合させない。同じ密集ページでタイトルとリストの両方へも付けない。一般的で誤解しにくい絵文字だけを使い、絵文字だけで意味を伝えず、PDF / PNGで欠け・白黒化・位置ずれがないか目視する
- **口語・内輪ノリの表現を使わない**（公開資料として社外の誰が読んでも大丈夫なトーンに）。例: バトル→競争、ミソ→重要なのは、刺さる→有効、腹をくくって→受け入れる、どっしり→腰を据えて。「コスパ・タイパ」など定着した語は可
- **強調は 2 段階で使い分ける**:
  - `**太字**` = 太字のみ（下線なし）。1 ページに数カ所まで
  - `***太字***` = 太字＋ゴールド下線（マーカー風。ブランドの署名的表現）。「ここだけ読めばいい」という**そのページの核心 1 フレーズ専用**で、1 ページ 1 カ所まで（表の中・lead・divider では使わない）
- **ページの結論と補足コールアウトを役割で分ける**:
  - `>`（blockquote）= そのページで発表者が言い切る中央1文（ラベル・背景・枠なし）
  - `<p class="note">` = 脚注（確認日、比較の前提、出典・資料の読み方。枠・背景・ラベルなし）
  - `<div class="callout good">` = GOOD（緑・推奨例や良い実践）
  - `.warn` = CAUTION（黄・適用条件、制約、導入前に確認すること）
  - `.danger` = WARNING（赤・禁止事項、重大なリスク、避ける行為）
  - `.info` = INFO（青・本文理解のため、その場で認識してほしい補足・背景情報）
  - 補足コールアウトのラベルは意味を瞬時に判別する共通マーカーなので、`GOOD / CAUTION / WARNING / INFO`の4種類で固定する。ページの結論にはラベルを付けない。`data-label`で「確認事項」「検証方法」「禁止」などの個別タイトルへ変更しない
  - div 内に Markdown を書くときは開始タグ・終了タグの前後に空行を挟む
- **takeawayはページを締める装飾ではない**: 本文から導ける判断があり、発表者がそのページで言い切る必要がある場合だけ使う。毎ページに置かず、タイトルや本文の繰り返し、注記、出典の読み方には使わない
- takeawayは**ラベルなし・中央揃えの1文・1行**に収める（目安は全角35文字前後）。背景・枠・角丸を付けず、本文より一段大きな文字と上下の余白で結論として区切る。結論文の幅に合わせたブロックを中央へ置き、直前の本文から離しすぎず、一つの結論として読める間隔にする。本文は濃紺で、核心語だけ`***太字***`のゴールド下線を引き、補足系コールアウトとは別の「ページの結論」として見せる
- **まとめページも結論を最後に置く**: 「まとめ」の直下へ結論を先に書かず、判断項目や要約を読ませたあと、ページ最下部へ`>`の結論文を置く。タイトルへ絵文字を付けず、「タイトル → 要約 → 結論」の順でページを閉じる
- **takeawayに句読点を使わない**: 本文ではなく見出し型のメッセージなので、文末の句点も途中の読点も置かない。2要素を区切る必要がある場合はダッシュ（`—`）を1つだけ使うか、句読点なしで読める語順へ直す
- **注記をコールアウト化しない**: 確認日、比較条件、資料の読み方、出典の前提など、読者の判断を変えず後から参照できればよい情報は`.note`で小さく置く。ただし投影時にも読める濃さを保ち、本文色より極端に薄くしない。青いINFOは、本文理解のため読者にその場で認識してほしい背景情報に限定する
- **表の補足は表の直後へ`.note`で置く**: テーマが中央揃えにし、可変幅の表でも注記だけがページ左端へ外れて見えないようにする。通常の注記は左揃えのままにする
- **表の内容を受ける箇条書きは `table-followup` で表とそろえる**: 表の直後に、その表から読み取れる利点・注意点などを2〜3項目置く場合は、ページへ `<!-- _class: table-followup -->` を付ける。テーマが表とリストを同じ幅で中央へ置き、箇条書きのマーカーを表の左端へそろえる。`center`と併用する場合は`<!-- _class: center table-followup -->`と書く。表と無関係な本文リストには使わない
- 2〜3カラムは`<div class="columns">`（3列は`columns three`）、対比カードは`<div class="card">`、KPIタイルは`<div class="stats">` + `<div class="stat">`
- **カードは独立した2〜3グループに限定する**: A/B対比や、境界を明確に分ける必要がある大きな概念だけに使う。同格の確認項目を要素ごとにカード化すると枠が増えて情報の流れを分断するため、3項目以上は通常の単一列リストにする
- **カードとコールアウトに装飾的な上線・横線・影を付けない**。薄い枠と余白だけでグループを示す。`box-shadow`はMarpのSVG／PDF変換で矩形の描画アーティファクトになることがあるため使わない
- **表は比較のために使う**: 2つ以上の製品・選択肢を同じ比較軸で横に見比べる場合、または「利用状況→推奨」のような正確な対応を探す場合に使う
- **表の直前へ説明文や結論文を置かない**: 「タイトル → 導入文 → 表」は、導入文と表のどちらから読めばよいか曖昧になる。表を使うページは原則「タイトル → 表」とし、読み方の補足は表直後の`.note`、判断や結論は表と補足を読んだ後の`>`へ置く。表の直前に通常段落またはblockquoteがあるページはレイアウト検査でNGにする
- **単一対象の特徴を表にしない**: 「強み｜内容」「確認項目｜理由」「向いている用途｜理由」のように、1つの製品・テーマの要素と説明を並べるだけなら、見出し付きリストか2階層の箇条書き（親 = 要素、子 = 説明）にする。この擬似的な2列表はリストより視線移動が増え、ページが硬く見える
- **見出し付き項目＋説明は `detail-list` を使う**: スライドへ `<!-- _class: detail-list -->` を付け、親項目を `- **🏷️ 観点名**`、説明をその直下のネストした `- 説明` として書く。テーマが子階層のマーカーを消し、説明だけを一段下げる。バックスラッシュ改行で親項目と説明を同じインデントへ並べない。3〜5項目すべてがこの二階層構造なのに`detail-list`がないページは、右側が過剰に空きやすいためレイアウト検査でNGにする
- **短い単一列リストは中央の読みカラムとして見せる**: 16:9で3〜5項目の短いリストをページ左端から始めると、右側だけが空いて視覚的な重心が崩れる。`detail-list`はテーマが約8割幅の中央カラムへ置き、項目間隔も確保する。同じページのGOOD / CAUTION / WARNING / INFO、脚注、結論文もそのカラム幅へそろえる。空きを埋める目的で左右2列へ分割したり、説明を右列へ置く擬似表へ変えたりしない
- **2分割は対等なA/B比較だけに使う**: 「できる／できない」「現状／将来」のように左右を同じ重さで読む場合に限る。主情報と補足、特徴と確認事項を左右へ分けると情報の階層と読む順番が崩れる
- **確認事項・注意・禁止・補足の帯は全幅で置く**: CAUTION、WARNING、INFO、GOODを`columns`の一方へ入れない。本文・リスト・比較の下へ全幅で置き、上から下へ自然に読ませる
- **単一製品は縦方向に読む**: 「向いている」特徴を全幅のリストで示し、その下に全幅の「向かない場合」または「確認事項」を置く。良い点と注意点を左右に分割しない
- **表でもリストでもない中間表現を作らない**: 観点名と説明を横2列・罫線だけで並べる擬似表は、比較表にも箇条書きにも見えず読み方が曖昧になる。単一テーマの3〜5要素は、`- **観点名**`の次行へ説明を置く2行構成の単一列リストにする
- **独立した箇条書きを裸の2カラムに分けない**: 読む順番が左右に散り、項目のまとまりも弱くなる。A/Bの明確な対比なら2枚のカード、順番に読む内容なら通常の単一列リストにする
- 表が連続するときは、内容を保ったままリスト、カード、Mermaidへ置き換えられるページがないか見直す。表と図を交互に置くこと自体を目的にせず、そのページで読者にさせたい行為が「比較・検索」か「理解」かで選ぶ
- **「A → B」「用語 — 説明」を1行に詰めた箇条書きにしない**: 正確な対応の列挙は表に、用語＋説明の列挙は2階層の箇条書きに分解する。1行詰め込みは折り返しで読みにくくなる
- 表の直後の余白はテーマが確保する（手動の空行調整は不要）

## 文章量バジェット（16:9・はみ出しの予防）

本文領域は約 **1024x592px、1 行あたり全角約 45 文字**（本文 22px）。16:9でも通常ページは左右128pxの中央カラムへ収め、横幅を情報量の増加に使わず、表・図・左右余白の可読性へ充てる。後から削るのではなく、書く時点で次に収める:

- **文字を小さくした分の余白は「空き」のまま残す**: v2.3 で本文を 25px → 22px に縮小した狙いは要素間の余白（呼吸）を作ること。空いたからといって箇条書きや文章を増やさない

- 1 ページ = **見出し + （表 or 箇条書き）+ コールアウト 1 個まで**。導入文は原則書かない（書くなら 1 行）
- 上限目安: **「箇条書き 4〜5 個 + コールアウト 1 個（3 行以内）」または「表 4〜5 行 + 締めの文 2 行」**
- **表セルは原則 1 行、意味を自然に区切る必要がある場合は 2 行まで**。自動折り返し任せで末尾に 1 文字だけ残さず、語句・列挙・意味のまとまりの境界へ `<br>` を置く。見出しセルと製品名は原則 1 行。3 行以上になる場合は文字を小さくせず、文章の短縮、列幅、表の分割・構造を見直す
- **blockquote / callout 内でハード改行しない**（1 つの `>` や div に文章を流し込む。幅が変わると改行位置も変わる）
- **強調記号（`**` / `***`）を約物（「」『』（）。、・）の直後で閉じない**: 例 `**エンジニア向けの「黒い画面」**` は閉じ `**` が `」` の直後に来るため Markdown が強調と認識せず、地の文に `**` がそのまま出る（markdown-it の右フランキング規則）。約物を強調の外に出すか、語順を変えて記号が通常文字に隣接するようにする（PNG 目視で `**` が生で残っていないか確認）
- 同じ主旨を導入文とコールアウトで**二重に言わない**
- タイトル（h1）の 2 行折り返しは禁止（全角 24〜26 文字を超えたら言い換えで短縮する）
- キーメッセージの複合語（「最善手」など）が行またぎで分断されたら、手動改行や言い回しで自然な位置に折る

## 検証ワークフロー

1. **機械検知**: `bun run ci`（typecheck + build + レイアウト検知）。`scripts/check-overflow.ts` が縦横のはみ出し、タイトルの 2 行化、表セルの 3 行化、表セル末尾の 1 文字残り、表直前の説明文、3〜5項目の二階層リストに`detail-list`がない横方向の偏りを NG ページ番号付きで出す。表セルの 2 行化自体はエラーにしない
2. **単体デッキの高速ループ**: 全デッキビルドを待たずに確認したいときは

   ```bash
   bunx marp slides/<デッキ>.md --images png --image-scale 1 -o <scratch>/deck.png --theme-set themes --allow-local-files --html
   ```

   でページ単位の PNG（deck.001.png…）を出し、該当ページだけ見る
3. **目視は絞る**: 通常編集では全ページ目視は不要。画像・挿絵を入れたページ、表セルへ意図的な `<br>` を入れたページ、折り返しが際どいページだけ PNG / `dist/slides/<デッキ>.pdf` で確認する。ただし 4:3 から 16:9 への移行時は、比率変更による重心・余白・改行の変化を見るため全ページをサムネイル一覧で一度確認する
4. あふれたら上記バジェットに沿って圧縮する（**意味・情報は削らない**。改行位置・冗長な語尾・二重表現の整理から）
5. **commit / push の前に、ビルド済み PDF をユーザー本人に目視確認してもらう**（機械検知と Claude の目視だけで完結させない）
   - `dist/slides/<デッキ>.pdf` の**フルパスをチャットに提示する**（ユーザーは VS Code などで自分で開くため、`open` コマンドで勝手に開かない）
   - ユーザーの OK が出てから commit / push の指示を待つ

## 挿絵・画像のポリシー

- **挿絵（手書き風インフォグラフィック）はユーザーが明示的に依頼したときだけ**入れる。こちらから「挿絵を入れましょうか」と積極提案しない。依頼されたら姉妹スキル `slide-infographic` に従う（画像生成の作法は `gallery-image-gen`）
- ロゴ・スクリーンショットなど挿絵以外の画像を指示されたときの Marp 記法は `references/image-patterns.md` を参照
- **スクショ・チャートなどの引用図版は `assets/slides/<デッキ slug>/` に置く**。`gallery/` には置かない（gallery はギャラリー一覧ページに自動掲載される ZENSHIN 制作画像専用。生成する挿絵は従来どおり gallery/）

## Mermaidの図

- **Mermaidは明示依頼がなくても使ってよい**。デッキを設計・編集するとき、表や箇条書きよりも関係性・順序・分岐が一目で伝わるページでは積極的に候補にする。これは生成画像の挿絵とは別扱いであり、「挿絵は明示依頼時のみ」の制約を受けない
- 特に次の場合はMermaidを優先して検討する:
  - Yes / Noや条件分岐から結論・推奨へ到達する
  - 3段階以上の工程、依存関係、データフローを示す
  - 1つの入力・判断が3つ以上の役割や処理へ波及する
  - 人、部署、システム間のやり取りや時系列を示す
  - 表が連続するデッキで、選び方・全体像・要点を1ページの図にして理解の入口と視覚的なリズムを作れる
- **表との使い分け**: 数値、機能の有無、製品ごとの正確な対応を比較・検索させるページは表を使う。読む順序、因果、分岐、全体構造を理解させるページはMermaidを使う。同じ情報を表と図で重複させない
- アクセントとして使う場合も、装飾だけの図にはしない。図を見ることで本文を読む前に構造や結論が分かることを採用条件にする。全ページを図にせず、デッキの理解を助ける要所へ置く
- **公開スライドではMermaidを実行時に読み込まない**。`.mmd`を正本としてSVGを事前生成し、Marpには通常の画像として埋め込む。CDNの`<script>`、`<div class="mermaid">`、カスタムengineは使わない
- `.mmd`、`mermaid-config.json`、生成済み`.svg`は`assets/slides/<デッキ slug>/`へ置く
- Mermaid CLIは版を固定する。標準コマンド:

  ```bash
  bunx @mermaid-js/mermaid-cli@11.16.0 \
    -i assets/slides/<デッキ slug>/<図>.mmd \
    -o assets/slides/<デッキ slug>/<図>.svg \
    -c assets/slides/<デッキ slug>/mermaid-config.json \
    -b white
  ```

- **Mermaid内は原則として標準テーマ＋白背景を使う**: スライドの紙色やブランド配色へ無理に合わせると、ノードと背景のコントラストが不足しやすい。図は情報構造の可読性を優先し、Mermaidの`default`テーマを起点にする。白背景には外枠や角丸を付けない
- **白背景の内側に余白を確保する**: `flowchart.diagramPadding`を40〜64px程度から調整し、端のノードや分岐ラベルが白領域へ接近しすぎないようにする。画像側へ枠やCSSのpaddingを足すのではなく、SVG生成時の余白として持たせる
- 16:9のYes / Noフローは原則`flowchart LR`、質問5個程度まで、1ノード2行以内にする。横長の本文領域を使い、図だけなら`w:960`〜`w:1000`、ページの結論文と同居する場合は`w:900`〜`w:960`を目安にする。到達点は質問ノードと形を分け、標準テーマの可読性を壊す独自配色を増やさない
- SVG生成後、対象ページをPNGとPDFで目視し、文字、矢印、分岐ラベル、余白を確認する
- MermaidなどのSVG図は内部に輪郭を持つため、写真用の角丸枠を付けない。スライド背景へ直接置き、図の中身と外枠を二重に囲わない
- 詳細と判断根拠は`docs/mermaid-in-marp.md`を参照する
